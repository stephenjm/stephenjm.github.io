From SDLC to ADLC: The Mindset Shift That Changes Everything

The problem with agentic software delivery isn't the AI. It's assuming the process can take care of itself. A personal experiment in what breaks when agents skip phase gates — and what that tells us about building reliable agentic workflows.

SJ
Stephen Senior Software Engineer · Security & Distributed Systems

Traditional SDLC vs Agentic SDLC — same phases, fundamentally different accountability model

I've been running a personal AI experiment for the past few weeks — building with agentic tools, documenting what breaks, and trying to understand where the process falls apart. What I found surprised me, not because the AI failed, but because of where things went wrong when it didn't.

"The AI isn't the weak link. The process is."

Every time an agent moved to the next phase without a clear, reviewed output from the previous one, the downstream work compounded the problem. Fast. What would have been a requirements gap in a traditional sprint became a design assumption, then a code smell, then a test that passed the wrong thing — all before I'd had a coffee.

What Traditional SDLC Gets Right

Before we talk about what changes in an agentic workflow, it's worth being precise about what doesn't. Traditional SDLC, done well, is rigorous for a reason.

Traditional SDLC

Human-driven · Sequential · Deliberate

Each phase gate is a natural forcing function — a moment where a human must review, approve, and take ownership before the work advances. This isn't bureaucracy. It's the mechanism that keeps errors small.

  • Human expertise at every gate
  • Deliberate, reviewed artefacts
  • Proven, battle-tested sequence
  • Defects are visible and attributable
Agentic SDLC (ADLC)

AI-orchestrated · Continuous · Self-documenting

The same phases, the same gate logic — but agents execute within each phase, generating artefacts autonomously. The pipeline can move at a speed no human team can match. Which is exactly why the guardrails matter more, not less.

  • Agents operate within human-defined constraints
  • Artefacts generated at every step
  • Feedback loops are automatic — but require design
  • Defects compound silently without gates

The shift from SDLC to ADLC is not a replacement of one with the other. It's an extension. Every principle that made your SDLC reliable — defined inputs, verifiable outputs, sequential gates, human accountability — becomes more important when an agent is doing the work, not less.

What the Experiment Revealed

In my experiment, the phases that failed weren't the ones where the AI generated bad output. They were the ones where I let the pipeline move forward before the previous phase was complete. The AI had no concept of "not yet." I had to provide that.

⚠️

The compounding problem

A missing context decision at the planning phase became an unstated assumption in requirements, which became an unreviewed constraint in design, which became an untested behaviour in code. Four phases, one root cause — and the agent never flagged any of it because it was too busy being confident.

This is the fundamental risk of agentic delivery: agents are optimistic by default. They complete tasks. They don't stop to tell you when a foundational decision is missing — they infer it, document the inference as fact, and carry it forward. The pipeline moves. The debt accumulates.

The Six Phase Gates That Cannot Be Skipped

Based on my experiment, here is how the six ADLC phases behave in practice — what the agent does, what the artefact looks like, and critically, what a human must own before the next phase begins.

1

Intent & Planning

The agent cannot define intent — it can only refine it. This phase must start with a human writing goals.md and constraints.md before any agent is invoked. The agent's role here is to surface gaps and ask clarifying questions, not to fill them with assumptions.

Gate Human defines
2

Requirements

The agent can draft requirements from the intent document, but every requirement that carries a risk or a constraint must be reviewed and signed off by a human. requirements.md and any Architecture Decision Records (ADRs) are the artefacts. Unreviewed requirements are the most dangerous single failure mode in ADLC.

Gate Human reviews
3

Design & Architecture

This is where the agent earns its keep — generating C4 diagrams, architecture markdown, and component boundaries from the reviewed requirements. But the outputs must be verified against the intent. A well-structured ARCHITECTURE.md and a set of C4 diagrams serve as the artefacts that constrain all downstream agent behaviour.

Gate Human approves
4

Code Generation

With reviewed architecture as the agent's ambient context — encoded in .github/instructions/ files — code generation is where full agentic autonomy is appropriate. The solution space is deterministic. The agent generates. CI validates. Pull request descriptions and commit messages become the artefacts of record.

Gate Agentic
5

Testing & Security

The agent can generate tests, run security scans, and produce test-report.md and security-scan.md. But a test that passes is only as valuable as the requirement it validates. If requirements were not reviewed in Phase 2, a passing test suite here is not evidence of correctness — it's evidence of internal consistency with a flawed spec.

Gate Human validates
6

Deploy & Monitor

Deployment can be fully automated. But the runbook, incident log template, and monitoring configuration must exist as human-reviewed artefacts before deployment runs. An agent that deploys without a runbook has not documented "done." It has just moved the problem to production.

Gate Human owns

Self-Documenting Requires a Human Trigger

One of the most appealing promises of ADLC is self-documentation — the idea that agents generate artefacts as a natural byproduct of their work. This is true, with one critical caveat.

💡

Self-documenting only works if you define "done" first

An agent will document what it did. It will not tell you whether what it did was correct, complete, or aligned with what you intended. The quality of the artefact is a direct function of the quality of the instruction. Garbage in, garbage out — except now it happens in seconds, not sprints, and the documentation looks authoritative.

In practice, this means the human role in ADLC is not to generate artefacts — it is to define the acceptance criteria that make a generated artefact trustworthy. Before an agent writes requirements.md, a human must have written what a good requirements document looks like for this system. Before an agent writes the architecture, a human must have defined the non-negotiable constraints that the architecture must satisfy.

The agent is a force multiplier. The human is still the source of intent.

The Mindset Shift in One Sentence

Traditional SDLC asks: "Did we complete each phase?"

Agentic SDLC asks: "Did a human define what done looks like before the agent started?"

That's the shift. It's not about removing human involvement — it's about moving it upstream. Less execution, more definition. Less review at the end, more constraint at the beginning. Design the guardrails before you let it run.

"The Toyota production line didn't get faster by removing inspection. It got faster by moving inspection to the right place."

If you're building agentic workflows today, start with this question at each phase: what artefact must exist before this agent starts, and who owns it? Get that answer in writing before the agent touches anything. The rest is automation.
